Odyssey - Cluster Finding

Once the image has been binarized, finding clusters of interesting pixels (1s on background of 0s) is a very well defined mathematical operation. Thus, it makes no sense to talk about cluster finding efficiency. Missing a cluster just signals the presence of a bug. 

The Odyssey clustering algorithm is the usual SySal one. It also correctly handles "A-shaped clusters" and "V-shaped clusters".
For each cluster, the returned information is:

Id, Area, X, Y, InertiaXX, InertiaXY, InertiaIYY.

Id is just a sequential order number.
Area is the number of pixels that make up this cluster.
X and Y are the cluster coordinates (measured from the center of the field of view in pixels).
InertiaXX, InertiaXY, InertiaYY: components of the matrix of inertia of that cluster. They are only three, but matrix of inertia is symmetric.

In order to speed up the clustering process, it can be split over several threads. The clustering speed is usually proportional to the number of threads used, until this number reaches the number of processors in the machine. Increasing the number of threads over the number of processors then just increases the overhead for managing multiple threads. Parallel clustering only works in asynchronous processing mode. In asynchronous mode, clusters for each sequence are produced after the next sequence starts, so there is 1 sequence delay.

Clustering needs the intervention of the host CPU. In synchronous mode, it must be so fast that it stays in sync with the acquisition, and it has 1 field delay with respect to grabbing end (2 field delay with respect to grabbing start). With fast cameras (>60 fps) this requirement is hardly met, and frames are missed, also because of OS thread time slicing (10 ms under Windows). Thus, for high performances, asynchronous operation is recommended. The synchronous functions have been implemented just for completeness and back-compatibility with older scanning modules.
